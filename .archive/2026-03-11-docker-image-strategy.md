# 决策记录：使用远程 Docker 镜像而非本地构建

**日期**: 2026-03-11
**决策者**: harryhua
**状态**: ✅ 已采纳

## 问题背景

model-converter 项目需要构建包含以下组件的 Docker 镜像：
- Python 3.11
- OpenGL 库 (libgl1-mesa-glx, libglib2.0-0)
- ML 库 (PyTorch 2.10, TensorFlow 2.21, Ultralytics 8.4)
- ST Edge AI 工具链
- ARM GCC 交叉编译器

### 初始方案：本地完整构建

尝试执行 `make build` 完整构建所有组件。

### 遇到的问题

1. **构建时间极长**
   - 下载 PyTorch: 32 分钟 (915.6 MB)
   - 下载 TensorFlow: 6 分钟 (572.4 MB)
   - 下载 CUDA 库: 20+ 分钟 (多个 200-700 MB 的库)
   - 总耗时：**超过 1.5 小时**

2. **磁盘空间占用巨大**
   - 构建过程中占用：**超过 30 GB**
   - 最终镜像大小：**23.1 GB**
   - 可用空间从 57 GB 降至 3.4 GB

3. **构建失败风险高**
   - 磁盘空间不足导致构建失败
   - 网络中断需要重新下载
   - Docker 存储损坏需要清理重建

## 考虑的替代方案

### 方案 1：继续本地完整构建
**优点**：
- 完全控制构建过程
- 可以定制化所有组件

**缺点**：
- ❌ 构建时间过长（1-2 小时）
- ❌ 占用大量磁盘空间（30GB+）
- ❌ 重复工作（每个开发者都需要构建）
- ❌ 维护成本高

### 方案 2：创建精简版镜像
**优点**：
- 镜像更小
- 构建更快

**缺点**：
- ❌ 需要额外开发和维护
- ❌ 可能缺少必要的工具链
- ❌ 仍需要下载大型 ML 库

### 方案 3：使用远程预构建镜像 ⭐️
**优点**：
- ✅ **首次部署快速**：20-30 分钟（主要是 Python 依赖）
- ✅ **节省磁盘空间**：共享基础镜像层
- ✅ **标准化环境**：所有开发者使用相同工具链
- ✅ **减少重复工作**：无需每个人都构建
- ✅ **可靠性强**：预测试的镜像

**缺点**：
- 依赖网络连接（首次拉取）
- 依赖 Docker Hub 可用性

## 最终决策

**采用方案 3：使用远程预构建镜像**

### 实施方案

1. **基础镜像**
   ```dockerfile
   FROM camthink/ne301-dev:latest
   ```
   - 包含 ST Edge AI、ARM GCC、系统工具
   - 大小：7.4 GB
   - 来源：Docker Hub

2. **增量构建**
   - 只安装 Python 3.11 和 OpenGL 库
   - 安装 Python ML 依赖（PyTorch, TensorFlow, Ultralytics）
   - 复制应用代码

3. **构建时间对比**

| 阶段 | 本地完整构建 | 使用远程镜像 |
|------|-------------|-------------|
| 基础工具 | 30-40 分钟 | **0** (预构建) |
| Python 依赖 | 20-30 分钟 | 15-20 分钟 |
| 应用代码 | < 1 分钟 | < 1 分钟 |
| **总计** | **50-70 分钟** | **15-20 分钟** |

4. **磁盘占用对比**

| 项目 | 本地完整构建 | 使用远程镜像 |
|------|-------------|-------------|
| 构建过程 | 30-40 GB | 10-15 GB |
| 最终镜像 | 23.1 GB | 23 GB (共享层) |
| 重复存储 | 每人一份 | **共享基础层** |

## 风险与缓解

### 风险 1：Docker Hub 可用性
**缓解**：
- 镜像拉取失败时，提供本地构建备选方案
- 文档中记录故障排查步骤

### 风险 2：网络速度慢
**缓解**：
- 提供镜像分层数据（展示需要下载的内容）
- 支持后台拉取，用户可并行处理其他事务

### 风险 3：基础镜像更新
**缓解**：
- 使用特定标签（`:latest` 可改为版本号）
- 定期测试并更新基础镜像版本

## 后续优化建议

1. **发布 backend 镜像到 Docker Hub**
   - 进一步减少首次部署时间到 5 分钟内
   - 用户只需 `docker pull` 即可使用

2. **创建多阶段构建**
   - 开发版本：包含完整工具链（当前方案）
   - 生产版本：只包含运行时依赖（更小）

3. **添加构建缓存**
   - 利用 Docker BuildKit 缓存
   - 加速重复构建

## 结论

使用远程预构建镜像方案显著提升了开发效率：
- **部署时间减少 70%**（从 50-70 分钟降至 15-20 分钟）
- **磁盘占用减少 50%**（构建过程从 30-40GB 降至 10-15GB）
- **开发体验改善**：标准化环境，减少重复工作

这是一个成功的架构决策，建议推广到其他类似项目。
